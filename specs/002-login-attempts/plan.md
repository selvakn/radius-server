# Implementation Plan: Login Attempt Log

**Branch**: `002-login-attempts` | **Date**: 2026-06-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/002-login-attempts/spec.md`

## Summary

Add a login attempt log to the RADIUS admin UI. Every authentication attempt is recorded in a new `auth_attempts` table. The admin sees a dedicated `/attempts` page showing up to 200 unique usernames with 24h count, last attempt time, and last outcome. Unknown usernames (not in the `users` table) show a one-click "add" button that pre-populates the new user form via query parameter.

## Implementation Phases

### Phase 1: DB Layer

1. Migration `003_auth_attempts.sql` — `auth_attempts` table
2. `internal/db/attempts.go` — `RecordAttempt`, `ListAttemptSummaries`, `PurgeOldAttempts`
3. Tests in `internal/db/attempts_test.go`

**Commit**: `Add auth_attempts table and DB layer`

### Phase 2: Record Attempts in Auth Handler

1. Call `db.RecordAttempt` in `auth.Handler.ServeRADIUS` after determining accept/reject
2. Start purge goroutine in `main.go` (run at startup + every hour)
3. Tests: verify attempt is recorded on accept and on reject

**Commit**: `Record auth attempts in RADIUS handler`

### Phase 3: Admin UI — Attempts Page

1. New `GET /attempts` handler — queries `ListAttemptSummaries`, renders `attempts.html`
2. Update `GET /users/new` to accept `?username=` query param and pre-fill the username field
3. `attempts.html` — table: username, 24h count, last time, last outcome, status badge, add button (unknown users only)
4. Add "attempts" link to nav bar
5. Tests: page returns 200, empty state, add button only for unknown usernames

**Commit**: `Add /attempts admin page with one-click user provisioning`
